# CLAUDE.md — Mandatory Implementation Contract

You are implementing the **AI-Assisted Bale Bot Project Builder Platform**.

You are not implementing a single bot project.
You are not implementing a fixed ticket system.
You are not implementing a demo-only app.
You are building a platform that can analyze project input, generate a project document, require human approval, produce a validated Bot Blueprint, and then generate a project-specific backend, User Bot, Admin Bot, Mini App/Web Panel, tests, docs, and Docker files.

## Absolute rules

### R1 — Builder Platform only
Do not hard-code a generated project.

The platform must support different generated projects. Project-specific User Bot commands, Admin Bot commands, APIs, database entities, pages, flows, roles, and permissions must come from an approved project document and validated Bot Blueprint.

### R2 — Documentation First gate
Implementation generation is forbidden unless the project has status `DOCUMENT_APPROVED` and a validated Blueprint.

No endpoint, service, worker, or internal function may generate project code directly from raw user text.

### R3 — CrewAI is not the deterministic generator
CrewAI may analyze, document, propose, review, and create Blueprint data. CrewAI must not be treated as the uncontrolled file generator.

Actual project files must be generated by a deterministic Generator module using validated Blueprint data and versioned templates.

### R4 — Templates are not fixed sample projects
Templates must be parameterized, modular, and selected from Blueprint features.

Templates must not contain hard-coded ticket, appointment, CRM, support, or admin business logic unless that logic is part of a named reusable module and activated by Blueprint.

### R5 — Shared backend, separate bots
Generated projects must support a multi-bot architecture:

- User Bot and Admin Bot have separate tokens.
- User Bot and Admin Bot have separate webhook paths.
- User Bot and Admin Bot have separate command/handler namespaces.
- They share backend, database, service layer, RBAC, audit, and frontend panel.

### R6 — Mini App equals Frontend Panel mode
Mini App is not a separate product. It is the same frontend panel running inside Bale WebView, with Bale-specific auth bootstrap. The same frontend must also be usable as a normal Web Panel outside Bale.

### R7 — Business logic belongs in backend services
Do not put business rules in bot handlers or frontend components. Bot and frontend are interfaces. Backend service layer is the authority.

### R8 — No SDK dependency as core Bale layer
Use a direct HTTP client abstraction for Bale API, preferably `httpx.AsyncClient`. External SDKs may be used only for prototype experiments, not as the core generated adapter.

### R9 — Validation before generation
Blueprint validation must run before any generator step. Invalid Blueprint must stop generation with actionable validation errors.

### R10 — Tests are mandatory output
Every generated backend module, bot webhook, Mini App auth route, permission rule, and generated API must have tests or explicit documented test stubs.

### R11 — No silent architectural changes
If a requested change conflicts with this contract, stop and report the conflict. Do not invent a new architecture.

### R12 — Phase discipline
Implement one phase at a time. Do not skip phases. Do not mix unrelated phases. At the end of each phase, report changed files, tests run, and remaining risks.

## Forbidden implementation patterns

Do not do any of the following:

- Do not generate project files directly from raw prompt text.
- Do not hard-code `ticket` as the only domain.
- Do not merge User Bot and Admin Bot into one bot unless Blueprint explicitly defines a single-bot project.
- Do not put Admin commands inside User Bot.
- Do not trust Mini App `initDataUnsafe` without backend verification.
- Do not bypass `DOCUMENT_APPROVED`.
- Do not store bot tokens in code.
- Do not implement generated project business logic inside frontend.
- Do not create untyped dynamic dictionaries when Pydantic schemas are required.
- Do not create endpoints without RBAC policy.
- Do not create admin operations without audit log.
- Do not leave generated files outside the expected repository structure.

## Required implementation style

- Python 3.12+
- FastAPI
- Pydantic v2
- SQLAlchemy 2.x
- Alembic
- PostgreSQL
- Redis
- Celery for long-running tasks
- Jinja2 for deterministic templates
- pytest for tests
- Ruff/Black/mypy compatibility
- React/Next.js or React+Vite with TypeScript for frontend templates

## Required response after each implementation phase

Claude must report:

1. What was implemented.
2. Files changed.
3. Tests added or updated.
4. Commands run.
5. Known gaps.
6. Whether the phase meets the acceptance criteria.

If tests cannot be run, Claude must explain why and provide exact commands to run.
